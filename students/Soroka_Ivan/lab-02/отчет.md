### Часть 3. Domain Layer (Доменный слой)

#### Доменные сущности

**Entity 1**: `Ticket`

```python
class Ticket:
    """Доменная модель: Тикет (Обращение)"""
    def __init__(self, ticket_id: str, title: str, description: str, category: str):
        self.id = ticket_id
        self.title = title
        self.description = description
        self.category = category
        self.status = "NEW"
        self.queue_id = None
    
    def assign_queue(self, queue_id: str):
        self.queue_id = queue_id
        self.status = "ASSIGNED"
```

**Доменные исключения**:
```python
class DomainException(Exception):
    pass

class InvalidTicketStatusException(DomainException):
    pass
```

#### Бизнес-правила
1. Тикет при создании всегда получает статус `NEW`.
2. При назначении исполнителя/очереди статус тикета должен измениться на `ASSIGNED`.
3. Нельзя закрыть тикет, если он не был назначен.

### Часть 4. Application Layer (Прикладной слой)

#### Входящие порты (Inbound Ports)

**CreateTicketUseCase**:
```python
from abc import ABC, abstractmethod
from dataclasses import dataclass

@dataclass
class CreateTicketCommand:
    title: str
    description: str
    category: str

class CreateTicketUseCase(ABC):
    @abstractmethod
    def create_ticket(self, command: CreateTicketCommand) -> str:
        pass
```

#### Исходящие порты (Outbound Ports)

**TicketRepository**:
```python
from abc import ABC, abstractmethod

class TicketRepository(ABC):
    @abstractmethod
    def save(self, ticket) -> None:
        pass
    
    @abstractmethod
    def find_by_id(self, ticket_id: str):
        pass
```

**NotificationPort**:
```python
from abc import ABC, abstractmethod

class NotificationPort(ABC):
    @abstractmethod
    def send_ticket_created_event(self, ticket_id: str) -> None:
        pass
```

### Часть 5. Infrastructure Layer (Инфраструктурный слой)

#### Входящий адаптер: REST API

**TicketController**:

```python
from fastapi import APIRouter, Depends
# ... импорты портов и DTO

router = APIRouter()

@router.post("/api/tickets")
def create_ticket(request: dict, service: CreateTicketUseCase = Depends(get_ticket_service)):
    command = CreateTicketCommand(**request)
    ticket_id = service.create_ticket(command)
    return {"ticketId": ticket_id}
```

**Пример запроса/ответа**:

```json
POST /api/tickets
{
  "title": "Не проходит оплата",
  "description": "Выдает ошибку 500 при вводе карты",
  "category": "Billing"
}

Ответ:
{
  "ticketId": "ticket-12345"
}
```

#### Исходящий адаптер: Repository

**InMemoryTicketRepository**:

```python
class InMemoryTicketRepository(TicketRepository):
    def __init__(self):
        self._db = {}
    
    def save(self, ticket):
        self._db[ticket.id] = ticket
    
    def find_by_id(self, ticket_id: str):
        return self._db.get(ticket_id)
```

**Принцип работы**:
Хранит данные в обычном Python-словаре (в оперативной памяти). Идеально подходит для локальной разработки и тестирования бизнес-логики без поднятия реальной БД.

### Часть 6. Dependency Injection (Конфигурация зависимостей)

**DependencyContainer**:

```python
class DependencyContainer:
    def __init__(self):
        self.ticket_repo = InMemoryTicketRepository()
        self.notification_adapter = DummyNotificationAdapter()
        
        self.ticket_service = TicketService(
            repository=self.ticket_repo,
            notification_port=self.notification_adapter
        )
    
    def get_ticket_service(self):
        return self.ticket_service
```

**Как работает DI**:
Контейнер создаёт экземпляры инфраструктурных адаптеров и внедряет (инжектит) их в `TicketService` через конструктор. `TicketService` воспринимает их просто как порты (абстракции).

## 5. Выводы

### Что получилось хорошо
Удалось чётко отделить бизнес-логику (`Ticket`, `Status`) от инфраструктуры. Слой `Domain` не имеет никаких зависимостей от веб-фреймворков (FastAPI) или ORM-аннотаций (SQLAlchemy). `TicketService` легко и невероятно быстро тестируется с помощью In-Memory заглушек.

### С какими трудностями столкнулись
Сложно было перестроиться с классической MVC/Django архитектуры на абстрактное мышление, где контроллер не вызывает базу данных напрямую, а всё общение идет строго через прослойку интерфейсов (портов) и инъекцию зависимостей.

### Что узнали нового
На практике освоен принцип инверсии зависимостей (Dependency Inversion Principle). Стало понятно, как гексагональная архитектура (Ports and Adapters) упрощает процесс рефакторинга: мы можем легко заменить локальную In-Memory БД на полноценный PostgreSQL, написав всего один новый класс-адаптер и поменяв одну строчку в DI-контейнере, вообще не трогая при этом бизнес-логику ядра.
